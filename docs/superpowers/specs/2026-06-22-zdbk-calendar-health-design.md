# ZJU Academic：ZDBK、学期转换与状态通知设计

日期：2026-06-22  
状态：设计已确认

## 背景

当前插件通过 `appservice.zju.edu.cn` 获取本科课表和考试。该接口对所有已测试学期返回 HTTP 504。Celechron 已在 2025 年 4 月停用这套实现，并改用 `zdbk.zju.edu.cn`。同一账号访问 ZDBK 时，CAS 登录成功，课表接口返回 HTTP 200 和 16 条原始记录。

现有刷新流程还存在两个问题：

- 一个数据源失败会中止全部刷新和提醒检查。
- 后台异常只写日志，绑定会话无法得知数据已经过期。

校历来自 `Xecades/zju-ical-py`。维护者在新学期安排发布后更新配置，插件无法提前获知可靠日期。因此，插件需要明确处理寒暑假、下学期安排尚未发布和人工日期填充。

## 目标

- 使用 ZDBK 获取本科课表和考试。
- 根据可靠校历完成学期转换，不推算未知日期。
- 各数据源独立刷新并保留最近一次成功数据。
- 将首次故障、持续故障和恢复状态通知所有绑定会话。
- 只在相关数据源异常时标注查询结果。
- 保留现有配置、绑定、提醒记录和对外工具名称。

## 本次不处理的内容

- 不开发 ZDBK 验证码填写页面。插件识别验证码要求并通知用户。
- 不引入成绩、主修、实践分等 Celechron 功能。
- 不根据往年日期推算新学期开学时间。

## 模块结构

### `ZdbkClient`

负责：

- 通过 ZJUAM 获取 `iPlanetDirectoryPro`。
- 通过 ZDBK SSO 获取 `JSESSIONID` 和 `route`。
- 请求课表和考试接口。
- 识别会话过期、验证码要求、超时、HTTP 错误和返回格式变化。
- 将 ZDBK 数据转换为插件统一数据结构。

课表使用：

- `https://zdbk.zju.edu.cn/jwglxt/kbcx/xskbcx_cxXsKb.html`
- 参数：`xnm`、`xqm`、`captcha_value`
- 解析字段包括 `kbList`、`kcb`、`djj`、`skcd`、`xqj`、`dsz`、`xxq`

考试使用 Celechron 当前的 ZDBK 考试接口和字段映射。插件不再调用 AppService 课表与考试接口。

### `CalendarResolver`

负责：

- 读取第三方校历、校历缓存和人工配置。
- 校验学年、日期顺序、学期顺序和日期重叠。
- 计算当前学期状态和查询所需学期。
- 将课程模板转换为具体日期，并应用调休安排。

### `RefreshCoordinator`

分别管理：

- 校历
- 课表
- 考试
- 学在浙大任务
- PTA 任务

一个来源失败时，协调器继续处理其他来源，随后继续执行提醒检查。

### `HealthNotifier`

负责保存数据源状态、判断是否需要发送消息，并向所有绑定会话发送经过整理的通知。

## 数据源状态

每个数据源保存：

- `status`：`healthy`、`failed` 或 `waiting_calendar`
- `last_attempt_at`
- `last_success_at`
- `last_error_code`
- `last_error_message`
- `failure_started_at`
- `next_retry_at`
- `last_notification_at`
- 最近一次成功数据

插件继续保留现有顶层缓存字段，读取旧版 `cache.json` 时自动转换。插件不会清空旧缓存、会话绑定或提醒记录。

## 学期转换

### 校历来源

1. 插件读取第三方维护的上一学年、当前学年和下一学年配置。
2. 插件拒绝开始日期晚于结束日期、学期重叠、学年格式错误或学期顺序错误的数据。
3. `manual_calendar_enabled` 默认为 `false`。
4. 启用人工配置后，每个学期必须同时填写开始和结束日期。插件不再使用固定 55 天计算结束日期。
5. 人工日期只填充第三方缺失的学期。
6. 第三方发布同一学期后，插件使用第三方数据替换人工临时配置。

插件保留现有 `manual_*_begin` 配置，并新增对应的 `manual_*_end` 配置。只有开始和结束日期都有效时，该人工学期才会参与解析。升级过程不会删除旧配置值。

### 学期状态

- `active`：当天位于某个学期日期区间内。
- `vacation`：当前学期已经结束，下一学期日期已知。
- `calendar_pending`：当前学期已经结束，下一学期日期仍未发布。

插件只请求 `active` 学期，以及与课程查询时间范围相交的已知下一学期。插件不再请求全部历史学期。

进入 `calendar_pending` 后，插件可以保存 ZDBK 返回的课程模板，但不生成具体上课日期，也不发送课程提醒。插件向绑定会话发送一次说明。第三方或人工日期可用后，插件重新生成课程事件并发送恢复通知。

## 刷新与缓存

- 每个数据源独立判断缓存有效期和重试时间。
- 刷新成功后，插件原子更新该来源的数据和 `last_success_at`。
- 刷新失败后，插件保留最近一次成功数据，不修改 `last_success_at`。
- 仍未过期的旧数据继续参与提醒。
- 用户查询异常来源时，结果包含数据可能过期、最后成功时间和当前异常说明。
- 数据恢复后，普通查询不显示状态提示。

接口请求失败后的自动重试间隔为 5 分钟、15 分钟、1 小时，之后每小时一次。`calendar_pending` 每 6 小时检查一次第三方校历。手动刷新允许立即请求一次，不受自动重试时间限制。

## 通知规则

通知发送给所有已绑定提醒的会话。手动刷新或查询的发起会话同时收到即时结果。

- `healthy` 变为 `failed` 或 `waiting_calendar`：立即发送一次。
- 相同异常持续存在：24 小时后每天发送一次。
- 异常恢复：发送一次恢复消息。
- 通知发送失败：保留待发送状态，后续重试。

消息包含：

- 受影响的数据类型
- 用户可理解的异常说明
- 最后成功时间
- 是否继续使用旧缓存
- 用户可以采取的操作

消息不包含账号、Cookie、完整堆栈或内部请求参数。日志继续保存维护所需的技术信息。

异常类别包括学校接口错误、登录或会话失效、验证码要求、返回格式变化、第三方校历不可用和下学期安排尚未发布。

## 测试

### ZDBK

- 登录成功并取得所需 Cookie。
- 会话过期后重新登录一次。
- 验证码要求转换为明确状态。
- 超时、HTTP 错误和格式变化不会覆盖成功缓存。
- 课表与考试样本可以转换为插件统一结构。

### 校历

- 秋冬、春夏和跨学年日期。
- 寒暑假且下一学期日期已知。
- 下一学期尚未发布。
- 人工开始和结束日期填充缺失学期。
- 只有开始日期或只有结束日期时拒绝该人工学期。
- 第三方配置替换人工临时配置。
- 重叠或无效日期被拒绝。

### 刷新与通知

- 一个数据源失败时其他来源继续更新。
- 异常期间继续使用最近成功数据。
- 首次故障、每日提醒和恢复通知各发送一次。
- 消息发送失败后可以重试。
- 正常查询不显示状态提示，异常查询显示最后成功时间。
- 旧版缓存与状态文件可以读取并转换。

## 发布

1. 在本地完成自动化测试。
2. 更新服务器前保存插件代码、配置、缓存和状态文件副本。
3. 使用现有账号执行一次 ZDBK 只读验证。
4. 更新插件并重新加载。
5. 强制刷新一次，检查课表、考试、DDL、通知和容器日志。
6. 验证失败时恢复更新前文件。

本次改动不增加第三方 Python 依赖。

## 验收条件

- AppService 故障不会影响课表获取。
- ZDBK 返回有效课表时，插件生成正确的课程日期和提醒。
- 下学期日期未知时，插件不生成推算课程，并向用户说明状态。
- 单个来源失败不会停止其他刷新或提醒。
- 异常期间保留旧缓存，查询结果显示过期提示；恢复后提示消失。
- 所有绑定会话收到首次故障、每日提醒和恢复通知。
- 服务器现有配置、绑定和提醒记录得到保留。
